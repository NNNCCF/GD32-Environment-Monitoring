# GD32 FreeRTOS 多传感器环境检测系统

## 项目简介

这是一个基于 GD32F10x 微控制器和 FreeRTOS 的多任务环境检测项目。系统集成了 DHT11 温湿度传感器、MQ2 气体传感器、OLED 显示屏、按键、LED 和串口通信，可用于基础环境数据采集与状态显示。

## 功能特性

- 基于 FreeRTOS 的多任务调度
- DHT11 温湿度采集
- MQ2 模拟量采集与百分比换算
- OLED 实时数据显示
- UART 数据输出与命令控制
- 按键控制 LED
- LED 周期闪烁
- 独立看门狗故障恢复

## 硬件要求

- GD32F10x 系列 MCU
- DHT11 温湿度传感器
- MQ2 气体传感器模块
- 0.96 寸 OLED 显示屏
- 按键
- LED
- USB 转串口模块

| DHT11 | DATA | PB8 | 温湿度数据线 |

| MQ2 | VCC | 5V | 模块供电 |

| OLED | GND | GND | 地 |

- 所有模块共地

## 运行说明

### 系统任务

当前工程包含以下主要任务：

- `vTaskMQ2`: 采集 MQ2 数据并通过串口输出
- `vTaskDHT11`: 采集温湿度并通过串口输出
- `vTaskOLED`: 刷新 OLED 显示内容
- `vTaskLED`: 周期闪烁 LED
- `vTaskKEY`: 扫描按键
- `vTaskUartCmd`: 接收串口命令并控制 LED
- `vTaskWDG`: 看门狗监管任务

### 串口参数

- 波特率: `115200`
- 数据位: `8`
- 停止位: `1`
- 校验位: `无`

### 串口命令

- `LED:ON`
- `LED:OFF`
- `LED:TOGGLE`

## 看门狗设计

### 当前实现

工程已经启用 GD32 独立看门狗 `FWDGT`，位于 `USER/main.c`。

当前不是“固定周期无条件喂狗”，而是“任务健康检查通过后才喂狗”：

- `MQ2`
- `LED`
- `KEY`
- `DHT11`
- `OLED`

这些关键任务都会周期性上报心跳。

`vTaskWDG` 会检查每个关键任务是否在规定时间内完成过一次心跳更新。只有全部任务都健康时，才调用 `fwdgt_counter_reload()` 喂狗。

如果某个关键任务卡死、长时间阻塞、跑飞或无法回到任务循环，心跳会超时，监管任务将停止喂狗，最终由硬件看门狗触发系统复位。

### 当前超时策略

看门狗硬件超时约为 8 秒，软件监管窗口按任务周期分别设置：

- MQ2: 3 秒
- LED: 4.5 秒
- KEY: 0.5 秒
- DHT11: 4.5 秒
- OLED: 2 秒

这样可以覆盖任务正常调度抖动，同时避免单个任务失效时系统继续被“假健康”喂狗。

### 设计价值

这种方式相比单独放一个高优先级喂狗任务更有意义，因为它能检测：

- 单个业务任务死循环
- 某个关键任务长期阻塞
- 调度异常导致任务不再按周期执行
- 系统整体卡死

但它不会把长期等待串口输入视为故障，所以 `vTaskUartCmd` 没有纳入强制心跳监管。

## 故障排查

### OLED 无显示

- 检查 `PB6/PB7` 接线是否正确
- 检查 OLED 供电是否正常

### DHT11 无数据

- 检查 `PB8` 接线
- 检查传感器供电

### MQ2 数据异常

- 检查 `PA0` 模拟输入接线
- 检查 MQ2 模块供电与预热状态

### 串口无输出

- 检查 `PA9/PA10` 接线
- 检查串口工具参数是否为 `115200 8N1`

### 系统频繁复位

- 先检查是否由看门狗触发
- 检查某个关键任务是否卡死或执行时间过长
- 调试时注意：如果没有配置调试冻结，看门狗在断点停住时仍可能继续计时并复位
